## Root Cause
The `qr_encoder_lite.py` script automatically selected the minimum possible QR version (up to 40) required for the provided data length without giving the user the capability to constrain or force an upper limit for the QR version.

## Solution
1. **Added Argument Parsing:** Added the `--max-version` argument via `argparse` to allow users to specify the maximum version (e.g., `--max-version 10`).
2. **Dynamic Version Selection:** Adjusted the `_select_version` function to take `max_version` as an upper limit for the iteration when scanning the error correction table. It now throws a `ValueError` appropriately if the data payload exceeds the capacity of the `max_version`.
3. **Dynamic Chunk Limit Adjustments:** Updated the `__main__` entry point to compute the maximum bytes capacity dynamically. If the given `chunk-size` exceeds the maximum capacity of `max_version`, the `chunk-size` is automatically restricted to ensure chunks encode successfully within `max_version`. Added helper function `_get_max_data_len`.
4. **Tested Changes:** Added local custom script to verify logic and ran `ut_test.sh` to confirm no regressions were introduced.
